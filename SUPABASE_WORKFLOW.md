# ⚡ Supabase Database Workflow (CLI & Migrations)

Этот документ описывает современный процесс управления базой данных в проекте Grocery App с использованием миграций и автоматической типизации.

## 1. Структура папки `supabase/`

- `migrations/` — **Единый источник правды**. Содержит SQL-файлы всех изменений структуры базы данных. Каждая миграция имеет временную метку и уникальное имя.
- `seed.sql` — Данные для первоначального наполнения (продукты, категории для тестов).
- `config.toml` — Конфигурация локального окружения Supabase.

## 2. Основные команды (Scripts)

Используйте сокращенные команды из `package.json`:

- `npm run supabase:types` — Обновить TypeScript-типы из облака (`types/supabase.ts`).
- `npm run supabase:pull` — Снять текущий слепок структуры из облака в новый файл миграции.
- `npm run supabase:push` — Отправить локальные миграции в облачную базу данных.
- `npm run supabase:status` — Посмотреть статус локального сервера или разницу с облаком.
- `npx supabase start` — Запустить локальное окружение Supabase (Docker).
- `npx supabase stop` — Остановить локальное окружение.
- `npx supabase db reset` — Полный сброс локальной базы и применение всех миграций заново.

## 3. Как вносить изменения (Процесс)

### Сценарий А: Вы меняете структуру в панели Supabase (Dashboard)
1. Вы внесли изменения (создали таблицу, колонку, изменили RLS).
2. Выполните: `npm run supabase:pull`.
3. CLI создаст SQL-файл в `supabase/migrations/`.
4. **Важно:** Сразу обновите типы: `npm run supabase:types`.

### Сценарий Б: Вы пишете миграцию вручную (Рекомендуется)
1. Создайте новую заготовку:
   ```bash
   npx supabase migration new name_of_migration
   ```
2. Напишите ваш SQL в созданном файле в папке `supabase/migrations/`.
3. Примените к облачной базе: `npm run supabase:push`.
4. Обновите типы: `npm run supabase:types`.

## 4. Локальная разработка (Docker)

Для безопасного тестирования изменений и работы без интернета используйте локальный сервер Supabase:

1.  **Запуск**: `npx supabase start`.
2.  **Локальный Dashboard (Studio)**: Доступен по адресу [http://localhost:54323](http://localhost:54323).
3.  **Синхронизация**: При запуске или через `npx supabase db reset` все миграции из папки `migrations/` применяются автоматически.
4.  **Конфигурация**: Для работы приложения с локальной базой используйте файл `.env.local`:
    - `EXPO_PUBLIC_SUPABASE_URL=http://127.0.0.1:54321`
    - `EXPO_PUBLIC_SUPABASE_ANON_KEY=...` (ключ из вывода `supabase status`)

## 5. Золотые правила

1.  **Никакого `schema.sql`**: Мы не пользуемся общим файлом схемы, история миграций — наш единственный ориентир.
2.  **Типизация — закон**: Сгенерированный файл `types/supabase.ts` никогда не редактируется вручную.
3.  **Commit миграций**: Все `.sql` файлы из папки `migrations/` должны быть закоммичены в Git.
