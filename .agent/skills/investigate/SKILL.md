# Investigate: Систематическая Отладка

## ⚠️ Главное правило

**Никаких правок без подтверждённой root cause.**

Если правка не устранила причину — немедленный откат. Не добавлять обходные пути поверх неизвестной проблемы.

---

## 🔄 4 Фазы

### Фаза 1: Root Cause Investigation

**Цель:** Точно воспроизвести и изолировать проблему. Код не трогать.

Действия:
1. Попросить пользователя описать: что сделал → что ожидал → что произошло
2. Получить полный stack trace / error message (не пересказ)
3. Определить минимальные шаги воспроизведения
4. Изолировать: всегда воспроизводится или иногда?

**Специфика проекта:**
- **RLS-ошибка Supabase** (`permission denied`) — проверять политику для конкретной роли через Supabase Studio → Authentication → Policies
- **Навигационный баг** (back() не работает, экран не найден) — проверять стек: в каком `_layout.tsx` зарегистрирован экран
- **Reanimated краш** — проверять worklet-окружение, нет ли доступа к JS-объектам внутри `runOnUI`
- **AsyncStorage race condition** — проверять порядок вызовов в `useEffect`, наличие cleanup

---

### Фаза 2: Pattern Analysis

**Цель:** Найти контекст и похожие случаи. Код не трогать.

Действия:
1. `git log --all --oneline --grep="<ключевое слово>"` — история изменений вокруг проблемы
2. Поискать похожие паттерны в codebase (Grep по симптому)
3. Проверить существующие тесты — описывают ли они это поведение?
4. Проверить связанные файлы: если сломан компонент → проверить хук → проверить стор → проверить API

**Вопрос к себе:** "Когда это последний раз работало? Что изменилось?"

---

### Фаза 3: Hypothesis Testing

**Цель:** Сформулировать одну гипотезу, проверить, подтвердить или опровергнуть.

Формат гипотезы:
```
Гипотеза: [конкретное утверждение о причине]
Проверка: [как проверить без изменения production-кода]
Результат: [подтверждено / опровергнуто]
```

**Правила:**
- Одна гипотеза за раз. Не перебирать наугад
- Предпочитать проверки через логирование / console.log перед правкой кода
- Если гипотеза опровергнута — сформулировать новую, не продолжать ту же
- Максимум 3 гипотезы. Если ни одна не подтвердилась — вернуться к Фазе 1 и собрать больше данных

---

### Фаза 4: Implementation

**Цель:** Устранить подтверждённую root cause минимальными изменениями.

**Правила:**
- Scope lock: не трогать >3 файлов без явного разрешения пользователя
- Atomic: одна правка = одна причина. Не объединять несвязанные исправления
- После правки — запустить `npm test` (AI Regression Cycle из testing/SKILL.md)
- Если тест упал — откат правки, вернуться к Фазе 3

**После исправления обязательно:**
1. Написать тест воспроизводящий баг (чтобы не вернулся)
2. Зафиксировать находку в коммит-сообщении: `fix(scope): описание — root cause: объяснение`

---

## 🚦 Scope Lock

Если исправление требует изменений в >3 файлах — СТОП. Сообщить пользователю:
```
⚠️ Scope creep: правка затрагивает [N] файлов.
Это может означать архитектурную проблему, а не точечный баг.
Предлагаю: [варианты локального решения]
Хочешь продолжить с полным scope?
```
